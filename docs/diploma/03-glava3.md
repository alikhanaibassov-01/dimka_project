# 3 РЕАЛИЗАЦИЯ ВЕБ-МАРКЕТПЛЕЙСА QAZMARKET

## 3.1 Используемые технологии и средства разработки

Реализация веб-маркетплейса выполнена на основе следующего стека технологий.

Серверная часть построена на платформе Node.js — среде выполнения JavaScript, основанной на движке V8 и событийно-ориентированной неблокирующей модели ввода-вывода. Выбор Node.js обусловлен единством языка на клиенте и сервере (JavaScript), богатой экосистемой пакетов npm и хорошей приспособленностью к задачам веб-API. В качестве веб-фреймворка используется Express — минималистичный и наиболее распространённый фреймворк для Node.js, предоставляющий маршрутизацию, цепочки middleware и удобную работу с запросами и ответами.

Хранение данных реализовано на встраиваемой реляционной СУБД SQLite через библиотеку better-sqlite3, предоставляющую синхронный API с подготовленными выражениями и транзакциями. Хэширование паролей выполняется библиотекой bcrypt. Управление сессиями — библиотекой express-session. Загрузка файлов — библиотекой multer; последующая обработка изображений (изменение размера, перекодирование в формат WebP) — библиотекой sharp.

Клиентская часть реализована на стандартных веб-технологиях: HTML5, CSS (фреймворк Tailwind CSS) и «чистый» JavaScript (ES2020+) без клиентских фреймворков. Tailwind CSS — утилитарный CSS-фреймворк, при котором оформление задаётся набором атомарных классов непосредственно в разметке; это ускоряет вёрстку и гарантирует визуальную консистентность.

Инструментальные средства разработки: редактор кода Cursor / Visual Studio Code; система контроля версий Git с удалённым репозиторием на GitHub (`https://github.com/alikhanaibassov-01/dimka_project`); менеджер пакетов npm; платформа облачного развёртывания Railway; браузерные инструменты разработчика (DevTools) для отладки клиентского кода и анализа сетевых запросов.

## 3.2 Реализация серверной части и REST API

Точка входа сервера — файл `server/index.js`. При старте выполняются следующие шаги. Сначала вызывается функция `ensureDb()` из `server/setup-db.js`: она проверяет существование файла базы данных и при необходимости создаёт его по схеме `data/schema.sql`, заполняет справочник категорий и создаёт (либо синхронизирует) учётную запись администратора на основании переменных окружения `ADMIN_EMAIL` и `ADMIN_PASSWORD`. Такой подход делает развёртывание идемпотентным: приложение можно запустить на «чистом» сервере без ручной подготовки базы.

Далее создаётся экземпляр Express и подключаются middleware: `express.json()` для разбора JSON-тел запросов; `express-session` с секретом из `SESSION_SECRET` и cookie с флагами httpOnly и sameSite=lax; `express.static('public')` для раздачи статических файлов фронтенда. Затем регистрируются маршрутизаторы: `/api/auth` → `routes/auth.js`, `/api/products` и `/api/categories` → `routes/products.js`, `/api/orders` → `routes/orders.js`, `/api/upload` → `routes/upload.js`. Сервер слушает порт из переменной окружения `PORT` (предоставляется платформой Railway) либо порт 3000 при локальной разработке.

Разграничение доступа реализовано в `server/middleware/auth.js` двумя функциями-middleware. Функция `requireAuth` проверяет наличие в сессии идентификатора пользователя и при его отсутствии возвращает ответ 401. Функция `requireAdmin` дополнительно проверяет, что роль пользователя — `admin`, иначе возвращает 403. Эти функции подключаются к защищённым маршрутам декларативно, например: `router.post('/', requireAdmin, createProduct)`. Таким образом, проверка прав не дублируется в обработчиках и не может быть случайно пропущена.

Работа с базой данных ведётся через подготовленные выражения better-sqlite3. Например, выборка каталога формируется динамически из условий фильтрации, но значения всегда передаются параметрами (`?`), что исключает SQL-инъекции. Создание заказа обёрнуто в транзакцию `db.transaction(...)`: вставка записи в `orders` и всех позиций в `order_items` выполняется атомарно — при любой ошибке изменения откатываются целиком.

## 3.3 Реализация каталога товаров, фильтрации и карточки продукта

Каталог реализован страницей `public/catalog.html` и скриптом `public/js/catalog.js`. При загрузке страницы скрипт запрашивает список категорий (`GET /api/categories`) и строит панель фильтров, затем запрашивает товары (`GET /api/products`) с учётом активных параметров: `category` (slug категории) и `q` (строка поиска по названию). Сервер формирует SQL-запрос с условиями `deleted = 0`, фильтром по `category_id` и поиском `LIKE` по трём языковым колонкам названия, поэтому поиск работает независимо от языка ввода.

Полученный массив товаров отрисовывается карточками: изображение, название на активном языке интерфейса, бейдж региона, цена в тенге с разделением разрядов и кнопка «В корзину». Карточки выложены адаптивной сеткой Tailwind: одна колонка на смартфоне, две на планшете, три-четыре на настольном экране. Если по выбранным фильтрам товары отсутствуют, выводится дружелюбное пустое состояние с предложением сбросить фильтры.

Детальная страница `public/product.html` принимает идентификатор товара в параметре URL (`?id=N`), запрашивает `GET /api/products/:id` и отображает крупное изображение, полное описание на текущем языке, цену, регион, производителя, индикатор наличия и элемент выбора количества с кнопкой добавления в корзину. При запросе несуществующего или удалённого товара сервер возвращает 404, а страница показывает соответствующее сообщение со ссылкой на каталог.

## 3.4 Реализация корзины и оформления заказа

Корзина реализована полностью на стороне клиента (модуль `public/js/cart.js`). Состояние корзины — массив объектов `{ productId, qty }` — сериализуется в localStorage браузера, что позволяет гостю собирать корзину без регистрации и сохранять её между визитами. Счётчик товаров в шапке обновляется при каждом изменении. Страница `cart.html` при открытии запрашивает у сервера актуальные данные товаров по идентификаторам из корзины, отображает построчно состав с возможностью изменить количество или удалить позицию и вычисляет итоговую сумму.

Оформление заказа реализовано страницей `checkout.html` и скриптом `public/js/checkout.js`. Форма содержит блоки: контактные данные (ФИО, телефон — для авторизованного клиента предзаполняются из профиля запросом `GET /api/auth/me`); способ получения (переключатель «Доставка / Самовывоз», функция `toggleDeliveryMode()` показывает либо поля города и адреса, либо блок пункта самовывоза с картой); способ оплаты (Kaspi / при получении); комментарий к заказу.

При подтверждении скрипт выполняет клиентскую валидацию обязательных полей, собирает полезную нагрузку (контакты, способ получения, способ оплаты, состав корзины в виде пар «id — количество») и отправляет POST `/api/orders`. Сервер повторно валидирует данные, загружает актуальные цены товаров из базы, рассчитывает сумму на своей стороне и в транзакции создаёт заказ с позициями, фиксируя `price_at_order`. В ответ возвращается номер заказа; клиент очищает корзину и перенаправляет покупателя на `order-success.html?id=N`, где отображаются номер заказа и, для Kaspi, платёжные реквизиты.

## 3.5 Реализация регистрации, авторизации и личного кабинета клиента

Регистрация (`register.html`, обработчик POST `/api/auth/register`) принимает email и пароль, проверяет корректность формата и уникальность email, хэширует пароль bcrypt и создаёт запись в `users` с ролью `client`, после чего сразу открывает сессию — пользователь оказывается авторизованным без отдельного входа.

Вход (`login.html`, POST `/api/auth/login`) находит пользователя по email и сверяет пароль через `bcrypt.compare`. Во избежание перечисления учётных записей при неверном email и неверном пароле возвращается одинаковое сообщение об ошибке. При успехе в сессии сохраняются идентификатор и роль пользователя. Выход (POST `/api/auth/logout`) уничтожает сессию.

Личный кабинет (`account.html`, скрипт `public/js/auth-pages.js`) при загрузке запрашивает `GET /api/auth/me`; неавторизованный пользователь перенаправляется на страницу входа. Кабинет содержит две секции. Секция «Профиль» — форма с полями «Фамилия», «Имя», «Телефон»; сохранение выполняется запросом PATCH `/api/auth/profile` с серверной валидацией. Секция «Мои заказы» — список заказов клиента (GET `/api/orders/mine`) с номером, датой, составом, суммой, способами оплаты и получения и текущим платёжным статусом, отображаемым цветными бейджами («Ожидает оплаты», «Оплачен», «Оплата при получении»).

Шапка сайта реагирует на состояние авторизации: гостю показывается кнопка «Войти», клиенту — ссылка на кабинет с именем, администратору — ссылки на панели управления.

## 3.6 Реализация административной панели

Административная панель состоит из четырёх страниц, доступ к которым контролируется и на клиенте (перенаправление при отсутствии прав), и на сервере (middleware `requireAdmin` на всех мутирующих маршрутах) — клиентская проверка отвечает за удобство, серверная за безопасность.

Вход администратора вынесен на отдельную страницу `admin-login.html`. Учётная запись администратора не создаётся через публичную регистрацию: она формируется при старте сервера из переменных окружения, причём `setup-db.js` синхронизирует пароль с актуальным значением `ADMIN_PASSWORD` при каждом запуске — это позволяет сменить пароль администратора простым изменением переменной окружения на Railway.

Добавление товара (`add-product.html`) — форма с полями: названия и описания на трёх языках, категория (выпадающий список из справочника), цена, регион, производитель, наличие и изображение. Изображение загружается отдельным запросом на `/api/upload` (см. 3.7), полученный путь подставляется в полезную нагрузку POST `/api/products`.

Управление товарами (`manage-products.html`) — таблица всех неудалённых товаров с миниатюрой, названием, категорией, ценой и действиями «Редактировать» и «Удалить». Удаление выполняется запросом DELETE `/api/products/:id` с предварительным подтверждением; на сервере выполняется мягкое удаление (`UPDATE products SET deleted = 1`), сохраняющее целостность исторических заказов.

Редактирование (`edit-product.html?id=N`) загружает текущие данные товара, предзаполняет форму и отправляет PUT `/api/products/:id`; изображение можно заменить либо оставить прежним.

Заказы (`admin-orders.html`) — список всех заказов (GET `/api/orders/admin`) от новых к старым с раскрытием состава, контактами покупателя, способами оплаты и получения. Для Kaspi-заказов в статусе «Ожидает оплаты» доступна кнопка «Подтвердить оплату» (PATCH `/api/orders/:id/payment`), переводящая заказ в статус «Оплачен».

## 3.7 Реализация загрузки и обработки изображений товаров

Загрузка изображений реализована маршрутом POST `/api/upload` (файл `server/routes/upload.js`), защищённым middleware `requireAdmin`. Приём файла выполняет multer с хранением в памяти и ограничениями: допускаются только MIME-типы изображений, размер ограничен несколькими мегабайтами.

Принятый файл обрабатывается библиотекой sharp: изображение приводится к ограниченной ширине с сохранением пропорций и перекодируется в формат WebP с настроенным качеством. Это решает сразу три задачи: единообразие размеров изображений в каталоге; существенное сокращение веса файлов (WebP обеспечивает на 25–35 % меньший размер по сравнению с JPEG при сопоставимом качестве), что ускоряет загрузку каталога; безопасность — принудительное перекодирование уничтожает любые посторонние данные, которые могли быть внедрены в исходный файл.

Результат сохраняется в каталог `public/uploads/` под уникальным именем, сформированным из метки времени и случайного суффикса (исключение коллизий и перезаписи). Клиенту возвращается относительный путь, который сохраняется в поле `image_url` товара. Раздача изображений происходит через стандартный механизм статики Express.

## 3.8 Реализация оплаты через Kaspi и оплаты при получении

В главе 1 было обосновано, что международные платёжные шлюзы недоступны для приёма платежей в Казахстане, поэтому в проекте реализована полуавтоматическая схема оплаты на основе переводов Kaspi, дополненная оплатой при получении.

При выборе способа «Kaspi» заказ создаётся с платёжным статусом `pending`. Страница подтверждения заказа отображает покупателю платёжную инструкцию: номер телефона получателя перевода (переменная окружения `KASPI_PHONE`), имя получателя (`KASPI_RECIPIENT`), точную сумму заказа и обязательный комментарий к переводу вида «QazMarket #N», где N — номер заказа. Рядом выводится QR-код со ссылкой на kaspi.kz: покупатель со смартфона сканирует код и попадает в приложение Kaspi. Комментарий с номером заказа является ключом сверки: администратор, увидев поступление в выписке Kaspi, находит заказ по номеру и подтверждает оплату в панели — заказ переходит в статус `paid`, что немедленно отражается и в кабинете клиента.

При выборе способа «Оплата при получении» заказ создаётся со статусом `cod`; денежный расчёт происходит при передаче товара (курьеру или в пункте самовывоза), и онлайн-подтверждение не требуется.

Достоинства реализованной схемы: нулевая стоимость подключения, отсутствие передачи платёжных данных через сайт (платёж целиком происходит в защищённом приложении банка), полная прослеживаемость по номеру заказа. Ограничение — ручное подтверждение администратором; автоматизация через Kaspi Merchant API отнесена к перспективам развития (3.11).

## 3.9 Реализация самовывоза и отображения точки выдачи на карте

Самовывоз реализован как альтернативный способ получения заказа. На странице оформления переключатель «Доставка / Самовывоз» управляется функцией `toggleDeliveryMode()` (файл `public/js/checkout.js`): при выборе самовывоза поля города и адреса скрываются и исключаются из валидации, а вместо них отображается информационный блок пункта выдачи.

Блок пункта выдачи (модуль `public/js/pickup-map.js`) содержит: адрес «г. Алматы, пр. Аль-Фараби, 93»; встроенную интерактивную карту OpenStreetMap (iframe-embed) с маркером на точке выдачи; кнопку «Открыть в 2GIS» с короткой ссылкой `https://go.2gis.com/Pr8me` — на смартфоне она открывает приложение 2GIS с возможностью построить маршрут до пункта выдачи.

Следует отметить инженерное решение, принятое в ходе реализации. Первоначально для отображения карты использовался официальный виджет 2GIS (DGWidgetLoader), однако тестирование выявило критический дефект: внешний скрипт виджета загружался нестабильно и в ряде случаев приводил к полной блокировке отрисовки страницы оформления («белый экран») именно в момент выбора самовывоза. Дефект был локализован, виджет удалён и заменён комбинацией «лёгкий OSM-embed + внешняя ссылка на 2GIS», не зависящей от доступности сторонних скриптов. Данный случай подробно разобран в главе 4 как пример найденного и устранённого дефекта.

В заказе с самовывозом поле `delivery_method` принимает значение `pickup`; администратор видит соответствующую пометку в панели заказов и готовит заказ к выдаче в пункте.

## 3.10 Развёртывание приложения на платформе Railway

Для публичного доступа приложение развёрнуто на облачной платформе Railway (модель PaaS — Platform as a Service). Выбор платформы обоснован: бесплатным стартовым тарифом, достаточным для учебного проекта; нативной поддержкой Node.js без написания Dockerfile; автоматическим развёртыванием из GitHub-репозитория; встроенным управлением переменными окружения и выдачей публичного HTTPS-домена.

Процесс развёртывания организован по схеме непрерывной доставки. Репозиторий проекта на GitHub подключён к сервису Railway; каждый push в ветку `main` автоматически запускает сборку (установка зависимостей `npm install`) и перезапуск сервиса командой `npm start`. Публичный адрес приложения: `https://dimkaproject-production.up.railway.app`. TLS-сертификат и терминация HTTPS обеспечиваются платформой автоматически.

Конфиденциальная конфигурация вынесена в переменные окружения Railway и не хранится в репозитории: `SESSION_SECRET` (секрет подписи сессионных cookie), `ADMIN_EMAIL` и `ADMIN_PASSWORD` (учётная запись администратора), `KASPI_PHONE` и `KASPI_RECIPIENT` (платёжные реквизиты), `BASE_URL` (публичный адрес). Порт прослушивания приложение получает из системной переменной `PORT`, устанавливаемой платформой.

Идемпотентная инициализация базы данных (функция `ensureDb()`, см. 3.2) обеспечивает работоспособность первого запуска на «чистом» окружении: схема создаётся, категории заполняются, администратор создаётся автоматически. Локальная разработка полностью повторяет производственную конфигурацию: `npm install`, файл `.env` с теми же переменными, `npm start`, адрес `http://localhost:3000`.

## 3.11 Перспективы дальнейшего развития проекта

В ходе работы определены направления развития, выходящие за рамки дипломного проекта, но органично продолжающие его архитектуру.

Первое направление — автоматизация платежей через Kaspi Pay / Kaspi Merchant API: автоматическое выставление счёта и автоматическая смена статуса оплаты без участия администратора. Реализованная полуавтоматическая схема спроектирована так, что переход потребует изменения только платёжного модуля.

Второе — многопользовательская модель продавцов: роль `seller`, кабинет производителя с собственным каталогом и заказами, модерация товаров администратором. Ролевая модель и схема БД допускают такое расширение добавлением роли и поля `seller_id` в таблицу товаров.

Третье — миграция СУБД на PostgreSQL при росте нагрузки: благодаря изоляции слоя данных и использованию стандартного SQL миграция сводится к замене драйвера и переносу схемы.

Четвёртое — пользовательские сервисы: отзывы и рейтинги товаров, email-уведомления о смене статуса заказа, восстановление пароля, избранное.

Пятое — эксплуатационные улучшения: панель аналитики продаж для администратора, журналирование действий, автоматическое резервное копирование базы данных, полнотекстовый поиск.
